ailang

A programming language built for AI, not humans.

Every language you know was shaped by human limits. ailang asks a different question: if the thing writing the program is an AI that thinks in structure, what should the program actually be?

01 — the premise

Built for human limits we don't have.

Keywords, indentation, tidy scrollable files — those exist because we get tired and forget. An AI doesn't. It has one cost that matters instead: attention. So we optimize for that, not for reading.

human languages assume…
  • Limited memory → names, comments, structure
  • Reading fatigue → whitespace, formatting
  • Cost ≈ keystrokes
  • Ambiguity, fixed later
  • Style is subjective
…an AI actually has
  • Effectively perfect recall of the whole program
  • No fatigue — density is fine
  • Cost ≈ attention — a different metric entirely
  • Emits a fully-checkable artifact in one go
  • One canonical form: diffable, cacheable, dedup-able
02 — the idea

A program is a graph you can fold.

The source of truth isn't text — it's typed building blocks ("nodes") wired together. A wire only connects if the types match. The compiler catches a broken connection before anything runs.

Then the move that makes it all work: any cluster of blocks folds into a single block. The AI only ever looks at the level it's working on. Everything else stays folded behind a hash and costs nothing until it's opened.

One mechanism, four wins: attention-efficiency, security at a glance, free deduplication, and infinite nesting.

five nodes → one fold → repeat
03 — what makes it different

Four ideas doing the heavy lifting.

i

Graph, not text

The canonical form is a content-addressed node-graph. No syntax, no style wars, byte-identical every time. Humans get diagrams — never the source.

ii

Folding is the engine

Collapse a subgraph into one node. That single act gives abstraction, attention-efficiency, dedup, and a security property — effects bubble up, so nothing hides three folds deep.

iii

Effects live in the type

A node that touches the network says so in its signature. Ordering comes from a linear token, so two side-effects can't run in an undefined order. Determinism becomes a typing rule.

iv

Trust the verifier

No human reads the code, so trust moves to a tiny audited verifier, machine-checkable proofs, and adversarial AI reviewers. You trust the math, not the author.

04 — how it's built

One AI directs. Others execute. A dumb harness judges.

The build process borrows ailang's own philosophy: never trust an output you haven't verified.

Controller · Claude → Executors · local models → Harness · compile + test → Reviewer · a different model ↺

A strong reasoning model owns architecture, writes tiny task specs, and writes the acceptance tests first. Cheaper local models write the code. The compiler and tests are the only judge of facts — nothing is accepted until it builds, passes, and survives a red-team by a different model family. Code is guilty until proven innocent.

05 — roadmap

Narrow first. Prove the model. Then grow.

P0

Foundations

Graph data model, type system, content-addressing.

P1

Walking skeleton

Effects, primitive nodes, fold/unfold, durable executor, graph → Rust → WASM. It runs in a browser.

P2

Safety

Capabilities, contracts, graph-level architecture checks, reviewer-in-CI.

P3

UI & apps

Web-first cross-platform demo — where we go beyond workflow-only prior art.

P4

Verification

Proof-carrying nodes; shrink the trusted compiler to something humans can verify.

P5

Ecosystem

The AI defines new node types in the language itself. FFI. A content-addressed registry.

status · honest version

Design is complete; implementation hasn't started. "Bug-free" is a fantasy and we won't pretend otherwise — what ailang actually delivers is memory-safe, type-safe, deterministic, contract-checked, and verifiable. That's the real, sellable claim. The codename is provisional. A public repository is coming.
